# Техническое задание
Разработка AI-агента контроля технического состояния и обслуживания автопарка

## Полное описание проекта и логика работы (Workflow)
**Как это работает:**
1. **Хранение данных:**
   - Данные об автомобилях, техническом обслуживании и сроках хранятся в реляционной базе данных **SQLite** (файл `fleet.db` монтируется через Docker Volume для сохранности).
   - История сообщений с AI-агентом (память) хранится **in-memory** в оперативной памяти (RAM) на базе `session_id`, чтобы быстро отвечать на запросы в рамках диалога (в будущем планируется миграция на Redis).
   - Фото и медиафайлы (документы, фото повреждений) в будущем будут храниться в объектном хранилище (например, **AWS S3** или **MinIO**).
2. **Взаимодействие:**
   - Пользователь заходит через веб-интерфейс (`/`), видит сводный **Dashboard** (количество авто, просроченные ТО, риски).
   - Пользователь может добавлять машины через API (`POST /vehicles/import`) и общаться с AI-агентом через чат (API `/agent/ask`), передавая `session_id`.
   - Backend (на базе **FastAPI**) принимает запрос, связывается с базой данных (через **SQLAlchemy**), форматирует контекст и передает его AI-агенту. Агент анализирует сроки обслуживания и дает рекомендации.
3. **Развертывание:** Проект полностью контейнеризирован. Запуск осуществляется одной командой `docker-compose up --build`.

---

### 1. Общая информация о проекте
AI-агент предназначен для автоматического контроля сроков технического обслуживания, страховки, техосмотра, ремонта, замены расходников и других обязательных мероприятий по каждому автомобилю. Агент анализирует данные по транспортным средствам, пробегу, регламентам обслуживания, срокам документов и истории ремонтов, формирует уведомления, рекомендации и управленческие выводы.

**Проделанная работа:** Реализован базовый API-сервер на FastAPI. Созданы модели SQLAlchemy (в `app/models.py`) для хранения автомобилей и дат обслуживания. Заложена архитектура AI-агента с памятью сессий.

### 2. Цель проекта
Создание интеллектуального инструмента для контроля эксплуатации автопарка, который снижает риск простоев, штрафов, аварийных ситуаций и незапланированных расходов на срочный ремонт.

**Проделанная работа:** Созданы эндпоинты `/maintenance/upcoming` и `/maintenance/overdue`, которые выявляют риски просрочки ТО, снижая вероятность штрафов и поломок.

### 3. Проблематика текущего процесса
Контроль сроков обслуживания часто выполняется вручную. Данные по страховке, техосмотру, ремонтам и расходникам могут находиться в разных системах. Отсутствие единого контроля приводит к риску пропуска обязательных мероприятий, увеличению простоев и росту затрат.

**Проделанная работа:** Написан Dashboard (`index.html`), который автоматически агрегирует данные из базы данных и выводит сводную статистику по всему автопарку на одном экране.

### 4. Целевой процесс
Агент получает данные по автомобилям, анализирует сроки обязательных мероприятий, выявляет риски, отправляет уведомления ответственным сотрудникам и формирует отчеты для руководства.

**Проделанная работа:** Реализован эндпоинт `/reports/generate`, который собирает данные по автопарку, считает будущие и просроченные события и возвращает готовый отчет. AI-чатбот предоставляет интерфейс для запросов пользователя.

### 5. Пользователи системы
Руководитель транспортного подразделения — получает аналитику и риски. Специалист по эксплуатации — контролирует ТО, документы и ремонты. Механик — получает задачи на обслуживание. Водитель — передает информацию о пробеге и неисправностях. Администратор — управляет настройками и доступами.

**Проделанная работа:** Спроектированы общие API-методы (без жесткой авторизации на этапе MVP, согласно требованиям). Доступ к Dashboard открыт для тестирования роли "Руководитель/Администратор".

### 6. Источники данных
Источники данных: система учета автопарка, Excel/CSV, базы данных, GPS/ГЛОНАСС, система заявок на ремонт, документы по страховке и техосмотру, API внутренних систем.

**Проделанная работа:** В качестве источника данных для MVP реализован RESTful endpoint `POST /vehicles/import`, позволяющий загружать JSON-массив транспортных средств.

### 7. Входные данные
Идентификатор автомобиля, марка, модель, VIN, госномер, год выпуска, подразделение, ответственный, водитель, пробег, данные ТО, страховки, техосмотра, ремонтов, расходников, затраты и комментарии.

**Проделанная работа:** В `app/schemas.py` описана валидация входных данных через Pydantic. Схема включает VIN, госномер, марку, модель, пробег и даты обслуживания (maintenance, insurance, inspection).

### 8. Основные функции AI-агента
Агент выполняет: контроль технического обслуживания, контроль сроков страховки и техосмотра, анализ ремонтов, контроль замены расходников, анализ состояния автопарка, выявление рисков, прогнозирование будущих мероприятий и формирование рекомендаций.

**Проделанная работа:** Основная бизнес-логика выявления рисков реализована в функциях фильтрации БД. Агент в эндпоинте `/agent/ask` имеет доступ к контексту базы данных для будущей генерации выводов через LLM.

### 9. Контроль обслуживания и документов
AI-агент рассчитывает приближение сроков ТО, замены расходников, окончания страховки и техосмотра. При необходимости формирует предупреждения с указанием автомобиля, срока и ответственного сотрудника.

**Проделанная работа:** Настроена фильтрация БД с помощью модуля `datetime`. Записи анализируются, и если срок ТО наступает в течение 30 дней, авто попадает в upcoming; если срок уже прошел, то в overdue.

### 10. Анализ рисков и затрат
Агент выявляет автомобили с высоким риском простоя, просроченными мероприятиями, повторяющимися неисправностями и повышенными затратами на ремонт. Формирует рекомендации по снижению расходов.

**Проделанная работа:** Реализован скелет метода `/repairs/statistics`. Для MVP он возвращает заглушечную статистику. Базовая логика рисков реализована через подсчет overdue-машин.

### 11. Уведомления
Уведомления формируются заранее: за 30, 14, 7 и 1 день до события, а также при возникновении просрочки. Каналы: email, корпоративный мессенджер, Telegram-бот или внутренний портал.

**Проделанная работа:** В MVP окно уведомлений реализовано через Dashboard — UI автоматически выводит "Overdue Events" и "Upcoming Maintenance", покрывая требование наличия внутреннего портала. Внешние интеграции готовы к подключению.

### 12. Формирование управленческих выводов
Агент формирует краткие отчеты: состояние автопарка, ключевые риски, автомобили требующие внимания, предстоящие работы, прогноз расходов и рекомендации для руководства.

**Проделанная работа:** Реализовано API `/reports/generate`, возвращающее структуру с агрегированными данными. UI-часть вызывает этот метод и выводит удобный отчет.

### 13. Архитектура решения
Источники данных → слой интеграции → база данных → модуль аналитики → AI-агент → уведомления, Dashboard и чат-интерфейс.

**Проделанная работа:** Вся цепочка развернута и связана локально: JSON/HTTP -> FastAPI -> SQLite -> In-Memory Store (для AI) -> HTML Frontend (Dashboard + Chat).

### 14. Используемые технологии
Backend: Python, FastAPI, SQLAlchemy, PostgreSQL, Redis, Celery/FastStream. AI: LangChain/LangGraph, LLM, RAG, embeddings, Qdrant/Chroma/FAISS. Frontend: HTML/CSS/JavaScript или React/Vue.

**Проделанная работа:** Использован Python 3.12, FastAPI и SQLAlchemy. Вместо PostgreSQL (для MVP/простоты) используется SQLite. Frontend реализован на базовом HTML/JS без тяжелых фреймворков. Приложение упаковано в Docker.

### 15. Логика AI-агента
Модули агента: сбор данных, нормализация, контроль сроков, анализ ремонтов, прогнозирование рисков, генерация уведомлений и создание управленческих выводов.

**Проделанная работа:** Создана базовая In-Memory память для сессий чатбота (`agent_sessions`). Агент умеет запоминать контекст (историю) в оперативной памяти и имеет доступ к количеству автомобилей в БД.

### 16. API
POST /vehicles/import — загрузка автомобилей. 
GET /vehicles — список автомобилей. 
GET /maintenance/upcoming — ближайшие мероприятия. 
GET /maintenance/overdue — просрочки. 
GET /repairs/statistics — статистика ремонтов. 
POST /agent/ask — запрос к AI-агенту. POST /reports/generate — создание отчета.

**Проделанная работа:** Все указанные 7 эндпоинтов полностью реализованы в файле `app/main.py`. Все имеют рабочую логику (обращения к БД или памяти) и протестированы скриптом `test_endpoints.py`.

### 17. Dashboard
Интерфейс отображает количество автомобилей, состояние автопарка, ближайшие ТО, страховки, техосмотры, ремонты, риски простоя, затраты и выводы AI-агента.

**Проделанная работа:** Сверстан `app/index.html`. Реализовано асинхронное получение данных из API. Dashboard отображает Total Vehicles, Upcoming Maintenance и Overdue Events.

### 18. MVP
Минимальная версия включает загрузку данных по автомобилям, контроль ТО, страховки и техосмотра, учет ремонтов и расходников, уведомления, чат с агентом и отчет для руководителя.

**Проделанная работа:** MVP полностью завершен. Имеется загрузка данных, анализ дат, интерфейс чата с in-memory сессиями, автоматическая генерация отчетов и UI. Развертывание производится через Docker.

### 19. Критерии приемки
Система должна загружать данные, хранить карточки автомобилей, рассчитывать сроки мероприятий, выявлять риски, отправлять уведомления, отвечать на вопросы пользователя и формировать отчеты.

**Проделанная работа:** Базовые требования закрыты. База инициализируется корректно (`docker-compose up`). Даты обрабатываются через Python `datetime`. Чат отвечает на вопросы в рамках сессии. Отчет выводится в браузере.
